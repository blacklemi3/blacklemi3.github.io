Systems

系统拆解

Robot / 双 MCU

机器人控制链路

这套系统可以理解为“一个负责思考和交互的大脑，加上一个负责执行动作的身体控制板”。ESP32-S3 处理语音、联网、屏幕、远程通道和指令调度；Hi3863 运动控制板接收指令，解析协议，再通过 PCA9685 控制 8 路舵机完成动作。

交互侧语音唤醒、ASR、LLM、TTS、双眼屏幕、飞书远程消息、Flutter App。
通信侧UART 帧协议、帧头、长度、命令、CRC、ACK/NACK、重试和状态回传。
运动侧PCA9685 舵机控制、动作队列、并发保护、姿态执行和异常处理。
  • 小白视角：用户说话或从 App 发指令，主控先把“想做什么”变成结构化动作命令，再交给运动板执行。
  • 工程视角：主控和运动板之间不直接传自然语言，而是用稳定的二进制协议，降低误触发和动作失控的风险。
  • 公开边界：可以公开协议思想、模拟器和示例动作；不公开真实固件、密钥、PCB、3D 文件和内部交接包。
Drivers / 外设

嵌入式外设驱动与试验箱案例

这部分不是单个设备，而是一组围绕 GPIO、I2C、PWM、UART、ADC 等基础接口展开的外设驱动和试验箱场景。价值在于把“能点亮一个模块”推进到“模块有统一接口、能组合、能在案例里跑起来”。

传感与输入AHT20、红外、超声波、键盘模块、ADC 采集、基础传感器读取。
显示与输出OLED 帧缓冲、数码管、WS2812 灯带、舵机、电机和房间设备控制。
语音与通信UART 语音识别、MP3/播报模块、串口协议解析和状态反馈。
  • Python 驱动更适合快速验证和教学式硬件抽象，C 案例更接近嵌入式设备上的真实控制逻辑。
  • 试验箱案例把外设驱动放进房间、灯光、门禁、环境检测、动作控制等组合场景里，避免停留在单点示例。
  • 公开版本更适合整理成通用接口、mock 硬件和可运行测试，而不是直接发布原始课程包或公司目录。
ArkTS / App

“喝啥”消费级轻产品 App

这是一个面向年轻消费场景的 HarmonyOS NEXT / ArkTS App，用抽卡和轻量成长系统解决“今天喝什么”的选择困难。它不是硬件项目，也不是儿童教育内容，而是一个围绕即时决策、趣味反馈和留存路径设计的小产品。

产品目标18-35 岁用户，3 秒内给出答案，打开到结果不超过 5 秒。
工程组织MVVM 分层，页面、组件、服务、模型、存储和测试分开维护。
规则系统概率、稀有度、保底、时间权重、历史权重、分享次数和本地持久化。
  • 用户看到的是一次轻松的抽卡结果，背后需要处理选项池、权重、重复控制、视觉反馈和历史记录。
  • 设计上更接近消费工具和娱乐化决策，不应该被归到儿童课程或单纯教学案例里。
  • 公开版本可以保留产品结构、状态流转、规则模拟和截图，不暴露未授权素材或上架相关私有信息。
Network / 回滚

网络变更与自动回滚

这类工作表面上是改配置，真正关键的是“远程改错了还能回来”。通过 VLAN Trunk 拆分办公网和业务网，并在变更脚本里加入备份、自动回滚、取消回滚和验证日志，让网络变更不依赖临场记忆。

变更前备份 network、dhcp、firewall 配置，记录当前接口、网段和访问路径。
变更中写入 VLAN、接口、DHCP、防火墙规则，并启动延迟回滚。
变更后验证 SSH、网关、业务网段、办公网段和日志，再取消回滚任务。
  • 小白视角：就像在远程改一把门锁，必须提前留一把能自动恢复的钥匙。
  • 工程视角：把高风险人工操作拆成可复查步骤，任何一步失败都能回到可访问状态。
  • 公开版本只说明流程和脚本结构，不公开真实 IP、设备账号、现场拓扑和内部配置文件。
Vision / Edge AI

边缘视觉训练与端侧部署

这条线把视觉模型从“训练工具里的结果”推进到“设备上能加载、能识别、能看日志”。围绕 K210 / Maix / CanMV / SentryMV 风格设备整理标注、标签、模型信息、启动脚本和端侧检测摘要。

数据侧VOC XML、图片尺寸、标注框、类别标签和训练数据目录。
部署侧labels.txt、model-info.json、boot.py、kmodel 文件和启动检查。
验证侧端侧检测结果摘要、类别分布、置信度统计和 Markdown 报告。
  • 公开仓库使用虚构样例，保留校验流程和报告形态，不暴露真实图片、真实模型权重和训练日志。
  • 这类工具的意义不是替代训练平台，而是补齐训练后到设备部署之间容易漏掉的检查项。
